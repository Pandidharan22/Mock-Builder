### Date: 2026-06-30
* **Objective:** Establish the AppModel schema contract in code: create the `mockbuilder` package, mirror the JSON Schema as Python dataclasses, and implement the validation gate.
* **Implementation Notes:**
  > Created the `mockbuilder/` package (`__init__.py`, `models.py`, `cli.py`).
  > `models.py` defines stdlib dataclasses mirroring every schema definition — `Meta`, `DesignTokens` (with `Typography`), `Entity` (with `EntityField`), `Component` (with `Prop`, `InteractiveElement`, `Action`, `Mutation`), `Screen` (with `Region`, `Variant`), `Flow` (with `FlowStep`), and the root `AppModel`. Each carries a `from_dict` classmethod for building typed objects from a validated payload.
  > `validate_app_model(data: dict) -> None` loads `app_model.schema.json` from disk and validates the dict with `jsonschema.validate`, raising `ValidationError` on the first violation.
  > `cli.py` uses `argparse` to expose `build <url> -o <out_dir>`, wired to a `run_build` stub that prints `Pipeline stubbed for <url> -> <out_dir>`.
* **Challenges & Solutions:**
  > The schema allows `additionalProperties` on `colors`, `spacing`, `radii`, and `shadows`. Stdlib dataclasses don't model open-ended key sets cleanly, so these are kept as plain `dict[str, str]` fields rather than forcing a rigid dataclass — fidelity to the contract over rigidity in the mirror.
  > **Architectural decision:** `app_model.schema.json` is the *single source of truth*. The dataclasses are a typed convenience mirror for the Python pipeline and are deliberately NOT used for validation — all contract checking goes through the JSON Schema via `validate_app_model`. This keeps one authoritative definition and avoids a second, drifting source of truth (and is why we chose stdlib dataclasses over Pydantic).
* **Verification:**

## Phase 1: Evidence Capture

### Date: 2026-06-30
* **Objective:** Stand up the crawler + recorder: a headless Playwright browser that visits a URL, captures a denoised structural snapshot of the DOM, derives a deterministic `state_hash`, and persists evidence (screenshot + actionable elements) to `evidence/`.
* **Implementation Notes:**
  > Created the virtualenv (`python -m venv .venv`), `pip install -e .`, and `playwright install chromium`.
  > `mockbuilder/crawler/dom.py` holds two browser-side captures run via `page.evaluate()`:
  >   - **DOM normalization** — clones `document.body`, removes all text nodes plus `<script>`/`<style>`/`<iframe>`/`<svg>` tags, and strips every attribute except `class`, `role`, and `data-testid`, returning the cleaned `outerHTML`.
  >   - **Element discovery** — queries `a, button, input, select, [role="button"], [role="tab"]` and returns `{tag, text, selector}` per element, where `selector` prefers `id`/`data-testid` and otherwise builds an `nth-of-type` path; `text` falls back through `innerText` → `value` → `aria-label`.
  > `mockbuilder/crawler/crawler.py` defines `async class Crawler` with `crawl(url, max_states=1)`: launches headless Chromium, `goto(url, wait_until="networkidle")`, runs both captures, hashes the normalized DOM with `hashlib.sha256().hexdigest()`, creates `evidence/` at the project root, and writes `evidence/{state_hash}.png` (full-page) and `evidence/{state_hash}_elements.json`. `max_states` is the seam for multi-state crawling later; only the landing state is captured now.
  > `cli.py`'s `build` command now calls `asyncio.run(Crawler().crawl(url))` instead of the stub. Added `evidence/` to `.gitignore`.
* **Challenges & Solutions:**
  > **Why hash the *structural* DOM?** Live apps are full of volatile noise — timestamps, ad/tracking markup, randomized class hashes, injected `<script>`/`<svg>`, A/B copy, per-request ids. If we hashed raw HTML, every reload would look like a brand-new state and the crawl graph would explode with near-duplicates. By first normalizing (drop text + noise tags, keep only the structurally meaningful attributes `class`/`role`/`data-testid`) and *then* hashing, two renders that are the *same screen* collapse to the *same* `state_hash`, while a genuinely different layout produces a different one. This makes the state hash the natural unique key (de-dupe, screenshot/elements filenames) and is what will keep multi-state crawling convergent rather than infinite.
  > The sandbox has no outbound network (`ERR_NAME_NOT_RESOLVED`), so end-to-end verification used a local `file://` HTML fixture instead of a public URL.
* **Verification:**
  > Crawled a local fixture containing noise (`<script>`, `<style>`, `<svg>`) and six actionable elements. Result: a stable 64-char `state_hash`, a full-page PNG, and an elements JSON with exactly 6 entries — `data-testid` selector used for the tagged button, `nth-of-type` paths for siblings, and the `aria-label` fallback correctly capturing the search input's label.

### Date: 2026-06-30
* **Objective:** Add the recorders (network fixtures + design tokens) and upgrade the crawler from a single capture to a breadth-first crawl over multiple states.
* **Implementation Notes:**
  > `mockbuilder/recorder/network.py` — `attach_network_listener(page, output_dir)` subscribes to `page.on("response")`, and for JSON-typed GET/POST responses writes the body to `output_dir/fixtures/{METHOD_URL}.json` (URL sanitized to a filesystem-safe, length-bounded key). Body reads are wrapped in try/except so a consumed/blocked stream degrades gracefully instead of aborting the crawl. Attached *before* navigation so no early payloads are missed.
  > `mockbuilder/recorder/design_tokens.py` — `harvest_tokens(page)` injects JS to read the computed `font-family`/`font-size`/`background`/`color` of `<body>` and the background-color of the structural landmarks (`header`/`nav`/`main`/`footer`), with a fallback that samples backgrounds from the first few generic `<div>`s. `save_tokens()` writes `evidence/design_tokens.json`.
  > `mockbuilder/crawler/crawler.py` — `crawl(url, max_states=3)` is now BFS. A queue holds *actions* (`{"url": base, "clicks": [selector, ...]}`); each state is reached by navigating to the base URL and replaying its click path on a fresh page (drift-free). New states (by normalized-DOM hash) get screenshot + elements + tokens persisted, then branch on up to 2 clickable elements (preferring `a`/`button`), each enqueued as `clicks + [selector]`.
* **Challenges & Solutions:**
  > **Network fixtures = the harness's seed database.** The generated mock is backend-free, so its data has to come from somewhere real. By recording the live app's actual JSON API responses to `fixtures/`, we capture authentic payloads (product lists, prices, categories) that the reasoning stage turns into the `entities[].seed` records in the AppModel and the generator emits as local data files. This is what makes the mock look real without a server — the seed is harvested, not invented.
  > **BFS + the structural hash prevents infinite loops.** A naive click-crawler loops forever: nav bars, "back home" links, and no-op buttons keep bouncing between a handful of screens. Because every state is keyed by the hash of its *normalized* DOM (§ Phase 1 first entry), revisiting a screen produces a hash already in `visited_states` and is silently dropped — so cycles collapse to a finite frontier and the crawl terminates. `max_states` is a second, hard ceiling on top of that.
* **Verification:**
  > Built three linked `file://` pages (home → products / about, each with a "back home" link) plus a no-op button on home. Crawl recorded 2 unique states (home depth 0, products depth 1); the no-op button and both back-home links hashed to the already-visited home state and were correctly *not* re-recorded — demonstrating the loop guard. `design_tokens.json` and the `fixtures/` dir were created (fixtures empty offline, since `file://` issues no JSON requests).

## Phase 2: Reasoning

### Date: 2026-06-30
* **Objective:** Stand up the LLM client and the caching layer for the reasoning stage — turn per-state crawl evidence into a schema-valid AppModel, deterministically and cheaply.
* **Implementation Notes:**
  > `mockbuilder/reasoning/cache.py` — `get_cached_model(state_hash)` / `save_cached_model(state_hash, model_dict)`, a disk cache at `evidence/{state_hash}_model.json`.
  > `mockbuilder/reasoning/prompts.py` — `SYSTEM_PROMPT = ""` (filled in the next step).
  > `mockbuilder/reasoning/reason.py` — `async synthesize_model(evidence_dir, state_hash) -> dict`: checks the cache first; on a miss reads `{state_hash}_elements.json` and base64-encodes `{state_hash}.png`, sends both (image + elements JSON + "raw JSON only" instruction) to the async Anthropic client, parses the reply with `json.loads`, validates it with `validate_app_model`, and on failure appends the validation error to the message history and retries (up to 3 attempts). Only a schema-valid model is cached and returned.
* **Challenges & Solutions:**
  > **Caching on `state_hash` = determinism + cost control.** The LLM call is the one non-deterministic, paid step in the pipeline. Keying its output by the crawl's normalized-DOM hash means an unchanged screen never triggers a second call — identical evidence always resolves to the same `{state_hash}_model.json`. The cache is what makes the *whole* pipeline reproducible end-to-end, which matters more here than any per-call sampling setting.
  > **Validate-retry loop for hallucination mitigation.** LLMs happily emit JSON that looks right but violates the contract (missing required key, wrong enum, bad `data-testid` pattern). Rather than trust the first response, every candidate is run through the JSON-Schema gate (`validate_app_model`); a failure isn't fatal — the exact `ValidationError` is fed back into the conversation and the model is asked to fix it, up to 3 tries. This turns the schema into an automated corrector, so downstream stages only ever see contract-valid models.
  > **Two deviations from the brief, forced by current APIs (2026-06):** (1) The requested `claude-3-5-sonnet-20240620` is retired (404); substituted its documented drop-in `claude-sonnet-5`, pinned in the `MODEL` constant for a one-line swap. (2) `temperature=0` is rejected (400) on current Sonnet/Opus models — sampling params were removed — so it is omitted; the `state_hash` cache already provides stronger, exact determinism than `temperature=0` ever did (which never guaranteed identical outputs anyway). Streaming via `messages.stream().get_final_message()` is used so the 32K `max_tokens` budget (needed for seed-data-heavy models) doesn't hit SDK HTTP timeouts.
* **Verification:**
  > Offline (no network in sandbox): confirmed the package imports cleanly, `MODEL == "claude-sonnet-5"`, `SYSTEM_PROMPT == ""`, the cache round-trips a dict through `save`/`get` and writes `{hash}_model.json`, and the code-fence-stripping helper handles both fenced and bare JSON. The live LLM path (image → validated model) requires network/credentials and will be exercised on a first real run.

### Date: 2026-07-03
* **Objective:** Convert referential integrity from a flaky prompt assertion into a **deterministic compilation gate** inside the reasoning loop, so dangling flow references can never survive.
* **Implementation Notes:**
  > Added `verify_graph_integrity(model) -> list[str]` to `reason.py`: it collects the defined `screens[].id` set and the defined component `interactiveElements[].testId` set, then walks every `flows[].steps[]` and returns a violation string for each `expectScreen` that isn't a defined screen and each `testId` that doesn't map to a defined element.
  > `{id}` placeholders are handled by `_testid_matches()`: a flow `testId` matches if it equals a component testId exactly (covers literal `login-btn` and literal `story-link-{id}`) or if it fits the pattern with `{id}` treated as a single `[A-Za-z0-9_-]+` segment (covers a resolved `story-link-5`).
  > Wired into the validate-retry loop **immediately after** `validate_app_model(candidate)`: if `verify_graph_integrity` returns violations, we `raise ValueError("\n".join(violations))`. The `except` tuple now includes `ValueError` (checked in `_format_error` *after* `json.JSONDecodeError`, which subclasses it), the rejection is printed to the terminal (`[reasoning] candidate rejected: ...`), and the exact violation text is appended to the message history so the model retries with a targeted fix.

## Phase 1′ (Pivot): Two-Track Re-Architecture — Deterministic Record Extraction

> Phases 1–4 produced a *buildable, offline, navigable, styled* harness — but on a text-only free-tier model the **data** was still fabricated. Root cause was structural, not a model weakness: the prompt literally instructs *"generate 4–8 rows of seed data,"* fusing two jobs the model is unequal at — STRUCTURE (judgment: what kind of app, what the repeating unit is) and DATA (transcription: the exact rows). This phase re-opens Phase 1 to split those tracks: the crawler extracts real records deterministically; the LLM decides structure only; the generator zips them. Numbered **1′** because it re-architects the crawler↔reasoning boundary, not because it precedes Phase 4. Full rationale in `PLAN_v2.md`; working state in `SESSION_HANDOFF.md`.

### Date: 2026-07-07 — The pivot: diagnosis + PLAN_v2
* **Objective:** Name the defect precisely and commit a corrected architecture before touching code, so the fix is a re-boundary and not a rebuild.
* **Diagnosis (settled, from on-disk evidence):** The crawler had already captured ~30 real HN stories; the model used 3 and fabricated `Story 4 … Story 8 / example.com` to satisfy the "generate rows" rule. That instructed fabrication is the visible "40% fake." Design tokens, evidence capture, the generator, and the verifier were all confirmed *fine* — do not rebuild.
* **The corrected architecture:** Move structured-data extraction *out* of the LLM and *into* the crawler. `CAPTURE` (crawler) now also emits `records[]`; `REASONING` (LLM) receives **one** sample record's typed shape + tokens and returns STRUCTURE only; `SYNTHESIS` (generator) fills `seed[]` from the real records. This also fixes the documented token-budget problem — the model no longer receives 30 rows, so the request shrinks by an order of magnitude.
* **De-risking:** Proven end-to-end by a standalone PoC (`repeating_extractor_poc.py`) before any pipeline change — one detector, zero app-specific selectors, correctly extracts all 5 HN stories AND all 6 shop products. The PoC is the blueprint for Step 1.
* **Verification:** `PLAN_v2.md` + `SESSION_HANDOFF.md` committed; PoC runs green on both fixtures and live HN. No production code touched yet — the point of this entry is that the diagnosis is settled and must not be re-litigated.

### Date: 2026-07-08 — Phase 1′ Step 1: `crawler/records.py` (app-agnostic record extractor)
* **Objective:** Productionize the PoC as a standalone, fixture-tested module — **not** wired into the crawler yet. Prove the detector in isolation first, exactly as the PoC did.
* **Implementation Notes:**
  > **The DATA track, in one module.** `crawler/records.py` owns record extraction and never talks to a model. `_DETECTOR_JS` (in-page) assigns every element a data-independent structural *signature* (subtree tag-shape, depth-capped), groups structurally-identical siblings, scores each group by `size * min(leaf_richness, 8)`, and merges an adjacent non-group sibling to handle split rows (HN's title-row + subtext-row). Deterministic Python `infer_role(field)` then types each leaf — `image / rank / vote / price / number / comment_count / score / age / domain / title / meta` — improving on the PoC's coarse `price_or_number` by splitting **price** (currency symbol present) from bare **number**.
  > **Typed return shape.** `Field` / `Record` / `ExtractionResult` dataclasses; `infer_role` is a pure, browser-free function. No app-specific selectors anywhere — a mechanical genericity guard asserts the source contains none of `athing / titleline / sitestr / card / subtext`.
* **Challenges & Solutions:**
  > **Isolation before integration.** Tests run the real detector against two hermetic fixtures copied into `tests/fixtures/` (`hn_fixture.html`, `shop_fixture.html`) so they don't depend on files outside the test dir. The gate is strict: anti-fabrication (extracted HN titles equal the 5 real ones exactly; no `example.com`, no `^Story \d+$`), determinism (two runs identical), and table-driven `infer_role` cases.
* **Verification:** All fixture + unit tests green. HN → 5 stories with `rank/vote/title/domain/score/author/age/comments`, author correctly typed `meta` (not `title`); shop → 6 products with `image/name/price/unit`. **Live HN → 30 records; `/jobs` → 28 records that shorten gracefully** (score/author/comment_count simply absent — no field-stealing, no corruption). One known quantity: HN rows carry a duplicated `domain` (the link renders twice); harmless, to be normalized downstream, not in the detector.

### Date: 2026-07-08 — Phase 1′ Step 2: async seam + crawl-loop wiring
* **Objective:** The crawler is async; `extract_records` was sync-only. Add an async entrypoint over a single shared transform, then wire extraction into the async crawl loop, emitting `records.json` per state — without changing the frozen algorithm.
* **Implementation Notes:**
  > **One pure transform, two thin seams.** Refactored so `build_result_from_raw(raw)` — pure, no Playwright/I/O/await — is the *single* definition of raw→dataclasses. `extract_records(page)` (sync) and `extract_records_async(page)` (async) each only evaluate `_DETECTOR_JS` and delegate to it, so the two paths can never drift. `_DETECTOR_JS`, `infer_role`, grouping, and the merge were untouched (verified byte-identical).
  > **Serialization boundary.** `ExtractionResult.to_dict()` is the `records.json` wire format — recurses through `Record`/`Field` and drops `None` optionals (no `href: null` / `src: null` noise). The dataclass is the in-memory API; nothing downstream reads it, only the dict.
  > **Crawler wiring + failure isolation.** In `_visit`, at the same point as screenshot/elements/tokens, the crawler calls `await extract_records_async(page)` and writes `evidence/<hash>_records.json`, mirroring the existing `_elements.json` file convention. Extraction can **never** abort a crawl: any exception logs ERROR and writes a distinguishable sentinel; a legitimately-empty page writes the same empty shape but logs only INFO. One INFO line per state.
* **Challenges & Solutions:**
  > **Async parity is the real gate.** A dedicated test asserts `extract_records_async == extract_records` on both fixtures; if they diverge at all, the seam isn't pure. The sync `browser` fixture keeps a Playwright asyncio loop live on the main thread, so the async test runs `asyncio.run` on its own thread to avoid a "loop already running" collision.
* **Verification:** 32→ green including the purity anchor (canned `raw` dict → exact `ExtractionResult`, browser-free) and async parity. Live HN crawl emits `<hash>_records.json` (count 30) beside the other evidence; `grep -c null` on it → 0. Legitimate-empty path writes the empty sentinel with no ERROR; monkeypatched crash writes the error sentinel and logs ERROR — both proven by wiring tests.

### Date: 2026-07-08 — Phase 1′ Step 3: ranked multi-collection + nested/absorption dedup
* **Objective:** A page can have several repeating regions (a category strip *and* a product grid). Return **all** qualifying collections ranked by score instead of one winner — a deliberate breaking change to `ExtractionResult`, made now while `records.json` has zero consumers. The extractor ranks; it never decides which collection is the "main entity" (that is the reasoning stage's job — no `primary` flag anywhere).
* **Implementation Notes:**
  > **New shape.** `ExtractionResult.collections: list[Collection]` (sorted by score desc; may be empty). `Collection` carries `rank` (a *position* after sorting, not a verdict), `score`, `signature`, `count`, `field_count`, `records`. `to_dict()` → `{"collections": [...]}`; the top-level `records/count/signature` fields are gone.
  > **Detector: keep every group, then dedup.** The JS now keeps every group with ≥3 members (scoring formula **unchanged**), builds records per-collection with the per-collection merge, and sorts by score descending (V8's stable sort keeps document order on ties, so `collections[0]` stays byte-identical to the old single-winner). Then it drops collections another already covers, by two criteria: **(1) nesting** — every member of A sits inside a member of B (a redundant per-item wrapper, or a grid inside a page-wrapper); **(2) merge-absorption (containment-aware)** — every member of A was folded into B's records by the adjacent-sibling merge, *including the sub-structure groups inside an absorbed sibling*. Keep the higher score; tie keeps the outer.
  > **Sentinels, generalized.** `{"collections": []}` = legitimate empty (INFO only); the crawler's crash path writes `{"collections": [], "error": true}` (ERROR logged). The crawler INFO line now reports collection count and per-collection `rank/count/score/signature`.
* **Challenges & Solutions:**
  > **Two honest deviations from the literal spec, both needed to hit the DoD.** (a) The nesting predicate is "each member of A inside *some* member of B," not "all of A inside a *single* member of B" — the literal single-member form is essentially untriggerable under structural-signature grouping (a member with an extra nested group would differ structurally from its siblings and not form a group). (b) Merge-absorption dedup was *added* beyond the spec's nesting-only rule: without it, HN yields 2+ collections (the subtext metadata rows — already merged into the stories — plus their `<td>`/`<span>` sub-groups). Both preserve the regression gate `collections[0] ≡ old winner`.
* **Verification:** 46 tests green. Regression anchor asserts `collections[0]` equals the frozen Step-2 winner records/roles/signature/count. New `multi_collection_fixture.html` (a 7-item strip + a 6-card grid with a redundant per-card wrapper) → exactly **2** collections: grid (rank 0, score 24, 6×4 fields, image+price) then strip (rank 1, score 7, 7×1). Dedup suppresses the wrapper's third group; `grep -rn primary records.py` empty; no `primary` key in `to_dict()`. **Live HN → exactly 1 collection, 30 records** (score 240) — the containment-aware absorption collapses HN's ~10 nested candidate groups into the single story collection.

### Date: 2026-07-10 — Step 4 pre-flight: the schema blocks structure-only output
* **Objective (blocker check, before writing Step 4):** Determine whether the reasoning stage can even *accept* a structure-only response once the "generate rows" instruction is removed. Answer needed because the validate⇄retry loop hard-fails on schema violations.
* **Finding — yes, `seed` is required and would reject every valid structure-only response.** `app_model.schema.json` marks the entity `"required": ["name", "fields", "seed"]` with `additionalProperties: false`, so an entity without `seed` fails `validate_app_model`. That raise is caught in `reason.py`'s loop and fed back as a retry hint, so a structure-only candidate would burn every retry on the same `'seed' is a required property` error and fail the build. The prompt (`prompts.py`) *also* still demands *"generate exactly 4 to 8 realistic rows of `seed`."* Both must change together in Step 4, or the loop rejects everything.
* **Open decision for Step 4 (deferred to the user):** whether `seed` stays in the entity schema as an **optional** property (so the generator can later inject records into the same AppModel object — note `additionalProperties: false` means the schema must still *allow* it) or is removed from the entity entirely, with seed data living only in `records.json` / a separate artifact. This determines the exact schema edit and where the generator writes seed. No code changed in this entry — it exists to record the constraint so Step 4's prompt is testable rather than dead-on-arrival.

### Date: 2026-07-10 — Phase 2′ Step 4: reasoning is now STRUCTURE-ONLY (the fabrication defect dies)

### Date: 2026-07-10 — Phase 2′ Step 4b: fix the cache key (content-address the reasoning inputs)
* **Objective:** Kill the cache-key bug flagged during Step 4 before any further prompt work. `cache.py` claimed to be "content-addressed" but keyed purely on `state_hash` (`{state_hash}_model.json`), so a prompt/payload change did NOT invalidate stale entries — a run against an evidence dir holding a pre-change `_model.json` would silently return the OLD (seed-bearing) model and mask the change. This is why Step 4 needed a manual `unlink` workaround.
* **Implementation Notes:**
  > **Key (`cache.py`).** Added `_inputs_hash(system_prompt, user_payload, model_name)` = first 16 hex of `sha256`, with a NUL separator between the three parts so `('ab','c')` can't collide with `('a','bc')`. Filename is now `{state_hash}_{inputs_hash}_model.json` — `state_hash` retained so entries stay greppable per crawl state. `get_cached_model`/`save_cached_model` take the three input strings and compute the key internally. Deterministic: identical inputs → identical key (no timestamps/randomness).
  > **Wiring (`reason.py`, key only — no reasoning change).** Moved the cache lookup to AFTER the messages are built (the key hashes the payload, so the payload must exist first), extracted `system_content = messages[0]["content"]` / `user_content = messages[1]["content"]` and passed them + `MODEL` into both cache calls, and moved `AsyncGroq()` construction below the lookup so a cache HIT still needs no API key. **Hash what's sent, not a copy:** the hashed strings are the exact `content` objects handed to `_create_completion` via `messages` — not an independently re-derived payload that could drift.
* **Challenges & Solutions:**
  > **Migration is a non-event, by design.** Old `{state_hash}_model.json` files are simply never looked up under the new scheme — they become inert rather than being read or migrated. That is correct: they were the hazard. A fresh run writes a new-keyed file alongside; the stale one is left untouched. Proven by `test_legacy_statehash_only_file_is_inert`, which plants a seed-bearing old-scheme file and asserts synthesize never reads it.
  > **The two properties must hold together.** Over-fixing into a cache that never hits would be as bad as the bug. Both are locked: HIT on identical inputs (`test_cache_hit_skips_model_call` — model called exactly once across two runs) and MISS on any input change (`test_prompt_change_misses_stale_cache` / `test_payload_change_misses_stale_cache` — a fresh call happens and the new model, not the stale one, is returned).
* **Verification:** New `tests/test_reasoning_cache.py` — 12 tests, all green; full suite 58 passed. ### Date: 2026-07-10 — Phase 3′ Step 7: role inference by residual + position (title is the free-text payload, defined negatively)
* **Objective:** Fix the deferred `infer_role` bug that 6b's render exposed — short titles ("GPT-5.6") mis-roled as `meta`, so ~1/6 of the HN front page rendered its headline in the gray metadata line. `records.py` deliberately unfrozen.
* **Why not length, and why not a positive signal:** Length is the wrong signal (bumping the threshold relocates the failure). But so is every POSITIVE structural signal — heading tag works on multi (h3), fails HN (bare <a>) and shop (<span>); external href works on HN link-posts, fails shop/multi (no hrefs); "first anchor" fails shop (title is a span). There is NO positive feature that means "title" across the three. The title is the free-text payload — the thing that ISN'T typed data — so it must be defined NEGATIVELY.
* **Implementation (two-pass + anchoring):**
  > **Two-pass inference.** PASS 1 (`infer_role`, pure per-leaf): assign the SPECIFIC roles; a leaf matching none returns `UNCLAIMED`. PASS 2 (`assign_record_roles`, record-level): among the unclaimed leaves, in DOCUMENT ORDER, the FIRST becomes `title`, the rest `meta`. Position works where length fails because document order already encodes headline-before-byline — "GPT-5.6" (7 chars) is the first unclaimed leaf and beats the longer author string. `title`/`meta` are no longer pass-1 outcomes; `build_result_from_raw` calls the record-level typer.
  > **Anchoring the specific patterns (the second half, forced by the render).** The first render of the two-pass alone reached 28/30: 2 rows promoted the AUTHOR to the headline, because pass-1's greedy `.search()` patterns stole the real title — `_AGE_RE` matched "second" in "No leap **second**…" and "year" in "…5-**year**-olds", so the title was claimed as `age` and the author became the first unclaimed leaf. That's a SEMANTIC bug (page looks right, lies) — worse than the fidelity bug we started with, so we did NOT ship 28/30. Fix: anchor the whole `.search()` class — `age`, `score`, `comment_count`, `price` — to WHOLE-TEXT (`fullmatch`): a leaf is a role only when the ENTIRE leaf IS the structured value, never when a title merely contains the keyword. Widened alternatives for known formats (age: "yesterday"/"today"/"just now"; score/comment: comma-thousands AND HN's no-comma "1345 points"; price: optional "/unit" suffix like "₹52/kg"). Widen by ADDING anchored alternatives, never by loosening.
* **The deliberate trade-off (handoff note):** anchoring makes pass 1 stricter, so more leaves fall through to pass 2. On HN (uniform formats) that's safe. On an arbitrary site, an unanticipated format (e.g. an absolute date for age) now falls through to `title`/`meta` — untyped, not corrupted. We chose "unusual formats go untyped" over "greedy patterns steal titles": graceful degradation (data still present, just untyped) is strictly better than the reverse (a title eaten by a greedy pattern, an author promoted to headline).
* **Verification — 30/30, rendered:** Re-roled the exact 6b-render content: 30/30 real (external-href) titles correctly `title`; the 3 short-title wins (GPT-5.6, Hy3, Muse Spark 1.1) preserved, the 2 regressions (leap-second, AI-tutor) recovered as titles with authors (ChrisArchitect, catalinvoss, logickkk1) back to `meta`. All 17 pass-1 table cases still pass; the 3 per-field title/meta cases were deleted and re-homed as record-level tests. Shop/multi unaffected. 93 total green.
* **Scope:** `records.py` (two-pass + anchored patterns) and `tests/test_records.py` only. No `reason.py`/`prompts.py`/schema/`inject.py`/generator/`cli.py` changes — the role-vocabulary change propagated to reasoning purely through the 4b content-addressed cache (a genuine miss), and to injection purely through the new records.json.
